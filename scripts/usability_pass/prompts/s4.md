## Stage 4 — Session bulk-close + auto-close cron (change 7)

Goal: `bash scripts/usability_pass/verify/s4.sh` passes. Read the gate FIRST. It mandates
exact names — these are the contract:

- Model method `dojo.class.session._cron_auto_close_sessions()`: finds sessions with
  `state='open'` whose `end_datetime` is more than GRACE minutes in the past (GRACE from
  `ir.config_parameter` key `dojo_core.session_auto_close_grace_minutes`, default 60),
  resolves every `pending` enrollment's attendance to `absent` (creating/updating the
  attendance log the same way the kiosk does), and sets the session to `done`.
- An `ir.cron` record named like `%auto-close%` (e.g. "Dojo: Auto-close ended sessions")
  running that method at least hourly.
- Kiosk bulk close: `/kiosk/instructor/session/close` accepts a new boolean param
  `mark_remaining_absent`. When true, pending attendees are marked absent and the close
  proceeds instead of returning the `pending_attendance` error. Requires `instructor_key`
  (S2). Add a matching confirm button in the kiosk SPA close flow ("Close & mark absent").

The gate proves behavior end-to-end via odoo shell: it creates a throwaway past session with a
pending enrollment using ONLY these fields — `template_id` (first existing template),
`start_datetime`, `end_datetime`, `state='open'` — then runs `_cron_auto_close_sessions()` and
asserts the session is `done` and the enrollment's attendance resolved to `absent`. If session
`create()` requires more than those fields, give them safe defaults so this construction works.

Notes:
- Attendance resolution must reuse the existing log/enrollment sync logic from
  `dojo_kiosk/models/dojo_kiosk_service.py` (or factor it onto the session/enrollment model) —
  do not invent a second attendance write path.
- Cron + config parameter live in `dojo_core` (or `dojo_classes` if session model is defined
  there — follow the model's home module). Upgrade it, restart web before gating.

Done = gate passes + commit.
