## Stage 3 — Demo dataset (DEMO_RESCUE.md Phase 3)

Goal: `bash scripts/demo_rescue/verify/s3.sh` passes. Read that gate script FIRST — it is the
exact specification of the dataset (member counts, Smiths, ranks, images, active+upcoming
sessions with instructor assigned, enrollments, onboarding records, subscriptions, kiosk PIN).

Build ONE idempotent seed script (e.g. `scripts/demo_rescue/seed/demo_data.py`, run via the
odoo-bin shell pattern) plus a thin shell wrapper. Requirements beyond the gate:

- Times are RELATIVE TO RUNTIME: active session started ~10 min ago and ends ~+50 min; upcoming
  session starts ~+10 min (inside the kiosk's 15-min window); one completed earlier today.
  The wrapper must be re-runnable right before the demo to re-center the times (move existing
  seeded sessions rather than duplicating).
- Every seeded session AND its template must reference instructor1's
  `dojo.instructor.profile` — the new record rules hide unassigned sessions from instructors.
- Surnames for the search demo: ≥2 distinct "X Smith", one "Smithson", one "Doe", one multi-word
  surname. demo1/demo2 members enrolled in the active session.
- Spread workflow states across members so kiosk badges/cards have something to show: one
  onboarding ~60%, one complete; one unsigned waiver; subscriptions in different states
  (active / paused or pending / credits exhausted) — create a minimal subscription plan if none
  exists; one open instructor task whose title CONTAINS the member's exact name (task matching is
  name-ilike); a trial lead booked into the active session IF the dojo_crm trial fields exist.
- Profile images: tiny generated placeholder PNGs (initials on a colored square) are fine —
  base64 them into `image_1920`.
- Kiosk config: active, PIN 123456. Print the kiosk URL at the end of the seed run.

Done = gate passes + commit.
